# AI Coding Instructions

## Communication

Respond **in the same language as the user's message**.
If the user writes in Japanese, reply in Japanese. If in English, reply in English.

---

## Required Checks After Every Code Change

After **any** code change, always run both steps in order:

```sh
npm run lint   # 1. Lint (must pass with no errors)
npm test       # 2. Test (must pass with no failures)
npm run build  # 3. Build (must complete without errors)
```

Do not submit changes that fail either check.

---

## Node.js Version

Always use the Node.js version specified in `.node-version`.
If your current version differs, check for a version manager (`nvm`, `fnm`, `volta`, etc.) and switch before running any commands:

```sh
# nvm
nvm use "$(cat .node-version)"
# fnm
fnm use
```

## Project Overview

`lgtm-action` is a GitHub Action that automatically posts an LGTM image as a comment
when a pull request review or issue comment matches a configured pattern (default: `LGTM`).

- **Supported events**: `issue_comment`, `pull_request_review`
- **Runtime**: Node.js as specified in `.node-version` (currently `24.x`), ESM (`"type": "module"`)
- **Built with**: `@vercel/ncc` → outputs `dist/index.js`

---

## Project Structure

```
src/             Source TypeScript files
  main.ts          Entry point; orchestrates the action flow

test/            Test files (mirrors src/ structure)
  util.ts          Shared test utilities
```

## TypeScript Conventions

- **Strict mode** is enabled via `@tsconfig/strictest`. Do not weaken or disable strict checks.
- All code must be **ESM**: use `import`/`export`; never use `require()`.
- When importing local modules, always include the `.ts` extension:
  ```ts
  // correct
  import { foo } from './foo.ts'
  // incorrect
  import { foo } from './foo'
  ```
- Prefer **named exports** over default exports.
- Prefer **`interface`** over `type` for object shapes.
- **Do not use non-ASCII characters** in identifiers (variable names, function names, etc.).
  Non-ASCII characters in comments are acceptable but may be removed in future releases.
- Do not use `any`; use concrete types or `unknown` with type narrowing.
- Do not suppress type errors with `@ts-ignore` or `@ts-expect-error` unless absolutely unavoidable;
  if used, always add a comment explaining why.

## Linting & Formatting

```sh
npm run lint   # Check for issues
npm run fix    # Auto-fix issues
```

Key rules enforced by oxlint (`.oxlintrc.jsonc`):

- No `any` types — `typescript/no-explicit-any`
- No `require()` — `typescript/no-require-imports`
- No `@ts-comment` suppressions — `typescript/ban-ts-comment`

Do not add disable comments (e.g., `// oxlint-disable`) unless unavoidable; always include an explanation.

## Testing

- Test files live in `test/` and mirror the source structure:
  `src/foo.ts` → `test/foo.test.ts`
- Use the **Node.js built-in `node:test`** module: `suite`, `test`, `mock`, `before`, `beforeEach`, etc.
- Use `mock.module()` for module-level mocking (see `test/main.test.ts` for an example).
- **Always add or update tests** when modifying source code. Do not submit changes without test coverage.
- Run tests with:
  ```sh
  npm test
  ```

## Build

- `dist/` is generated by `@vercel/ncc` via `npm run build`.
- **Never edit files under `dist/` directly.** It is a build artifact that is automatically generated by CI/CD on the release branch; any manual changes will be overwritten.
- On `main`, `dist/` is listed in `.gitignore` and is not tracked in version control.

## Commit Conventions

Follow the [Angular commit message format](https://github.com/angular/angular.js/blob/master/DEVELOPERS.md#type).

```
<type>(<scope>): <summary>

<optional body>
```

| Type       | When to use                            |
| ---------- | -------------------------------------- |
| `feat`     | A new feature                          |
| `fix`      | A bug fix                              |
| `docs`     | Documentation only changes             |
| `style`    | Whitespace/formatting, no logic change |
| `refactor` | Code change with no feature or bug fix |
| `perf`     | Performance improvement                |
| `test`     | Adding or correcting tests             |
| `chore`    | Build process or tooling changes       |

Scope is optional; use it to indicate the affected area (e.g., `feat(event): ...`).

## Pull Requests

- The title should clearly summarize the change.
- Reference related issues in the description (e.g., `ref #99`).
- Mark in-progress PRs with a `[WIP]` prefix and open as Draft; remove both when ready for review.
- Resolve conflicts with `main` before removing `[WIP]`.
